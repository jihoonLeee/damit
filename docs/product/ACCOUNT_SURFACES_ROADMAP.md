# ACCOUNT SURFACES ROADMAP

## PM 결론

`로그인`, `회원가입`, `사장님 마이페이지`, `시스템 관리자 기능`은 모두 필요하다.

다만 한 묶음으로 만들면 안 된다.

- `로그인 / 회원가입 / 사장님 마이페이지`는 고객이 직접 쓰는 제품 표면
- `시스템 관리자 기능`은 내부 운영 표면

즉, 제품 정보구조상 같은 레벨이 아니라 **두 개의 다른 트랙**으로 봐야 한다.

---

## 왜 필요한가

### 1. 로그인

지금도 세션 기반 로그인은 있지만, 제품 관점에서 보면 아직 “운영형 진입 흐름”에 가깝다.

실제 서비스가 되려면 사용자는 아래를 자연스럽게 이해해야 한다.

- 어디서 시작하는지
- 로그인과 가입의 차이가 무엇인지
- 처음 들어오면 무엇을 해야 하는지

### 2. 회원가입

B2B 서비스라도 사장님이 스스로 시작할 수 있어야 한다.

현재 구조상 첫 로그인 후 회사 생성이 이미 회원가입의 일부 역할을 하고 있으므로,
완전히 별도 인증 체계를 만들기보다 아래처럼 정리하는 것이 좋다.

- `시작하기` 버튼
- 이메일 입력
- 매직 링크 인증
- 첫 진입 시 `이름 + 회사 이름` 입력
- 계정/회사 생성 완료

즉, **회원가입은 별도 비밀번호 폼이 아니라 “첫 로그인 온보딩”으로 구현**하는 것이 현재 제품과 가장 잘 맞는다.

### 3. 사장님 마이페이지

실제 운영에서는 단순 홈보다 “내 계정 / 내 회사 / 내 팀 / 내 설정”이 분명해야 한다.

사장님 입장에서 필요한 정보는 대체로 아래와 같다.

- 내 이름 / 이메일
- 현재 활성 회사
- 팀원/초대 현황
- 회사 기본 정보
- 로그인 및 세션 상태
- 보안 설정 상태
- 향후 요금제/결제 정보가 들어갈 자리

즉, 현재 `/home`은 허브 역할은 하고 있지만, **사장님 마이페이지로 보기엔 아직 업무 허브와 계정 허브가 섞여 있다.**

### 4. 시스템 관리자 기능

이건 꼭 필요하지만, 절대 일반 제품 내비게이션과 섞이면 안 된다.

시스템 관리자 기능은 아래처럼 완전히 내부용이어야 한다.

- 회사 검색
- 사용자 검색
- 로그인 전달/세션 이상 징후 확인
- 고객 확인 링크 상태 확인
- 감사 로그/Audit 조회
- 장애 대응 / 백업 / 복구 확인

현재 `/ops`는 owner 운영 콘솔 성격과 시스템 운영 콘솔 성격이 일부 섞여 있다.
장기적으로는 이 둘을 분리해야 한다.

## 추천 구조

### A. 고객용 제품 표면

1. `/`
- 랜딩

2. `/login`
- 이메일 로그인
- 초대 링크 수락
- 첫 로그인 유도

3. `/start`
- 사장님 시작하기
- 첫 로그인/첫 회사 생성 안내
- 사실상 가입 입구

4. `/home`
- 운영 허브
- 지금 어디로 가야 하는지 안내

5. `/account`
- 사장님 마이페이지
- 내 계정, 내 회사, 멤버/초대, 보안/세션, 회사 설정

6. `/app`
- 실제 업무 워크스페이스

### B. 내부 운영 표면

1. `/ops`
- 사장님 owner 운영 콘솔
- 내 회사 기준 운영 상태 확인

2. `/admin`
- 시스템 관리자 전용
- 다중 회사, 사용자, 로그인, 감사 로그, 복구/지원 도구

## 우선순위

### P0

- 로그인 흐름 정리
- 가입 입구(`시작하기`) 명시
- 첫 로그인 온보딩 명확화
- `/home`과 `/app` 역할 분리

### P1

- 사장님 마이페이지(`/account`)
- 팀/초대/회사 설정/세션 상태 UI
- owner가 자기 회사를 관리하는 흐름 정리

### P2

- 시스템 관리자 전용 `/admin`
- `/ops`와 `/admin` 경계 분리
- 전사 단위 지원/운영 도구 추가

## PM 판단

지금 가장 먼저 필요한 것은 아래 두 가지다.

1. `회원가입을 제품 표면에서 분명하게 보이게 만들기`
2. `사장님용 마이페이지를 운영 허브와 분리해서 설계하기`

즉, 다음 배치의 핵심은:

- `시작하기 / 가입 흐름`
- `사장님 마이페이지`

이고, 시스템 관리자 기능은 그 다음이다.
