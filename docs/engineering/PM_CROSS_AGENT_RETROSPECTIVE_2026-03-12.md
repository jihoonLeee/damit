# PM Cross-Agent Retrospective (2026-03-12)

## 목적

- `PM`, `Feature`, `Builder`, `QA`가 지금까지의 진행을 함께 되돌아본다.
- 무엇이 잘됐는지, 무엇이 아쉬웠는지, 앞으로 무엇을 유지/수정/중단할지 정리한다.

## What Went Well

### PM

- `큰 전환을 잘게 나누는 방식`이 효과적이었다.
- Fly Managed Postgres 같은 성급한 비용 지출을 피하고, Supabase Free + SQLite 유지 전략으로 더 현실적인 진도를 만들었다.
- 읽기와 쓰기를 분리해서 통합한 순서가 좋았다.

### Feature

- 제품 핵심 가치가 흐려지지 않았다.
- 매번 새로운 기능을 붙일 때도 `설명`, `기록`, `합의`라는 중심이 유지됐다.
- 브랜드 `다밋`도 제품 방향과 크게 충돌하지 않았다.

### Builder

- 실제 배포, staging, 시각 검수, Supabase preflight, migration 적용까지 끌고 갔다.
- repository abstraction 을 문서 수준이 아니라 실제 app route 에 연결하기 시작했다.
- 테스트가 계속 green 인 상태를 유지한 점이 좋다.

### QA

- regression culture 가 초반부터 잡혔다.
- route integration 마다 foundation, auth, runtime, repository 테스트를 같이 돌린 점이 안정감을 만들었다.
- staging smoke check 습관이 매우 좋았다.

## What Was Hard

### PM

- owner-token 파일럿 경로와 production beta 경로가 동시에 존재해 판단이 자주 복잡해졌다.
- `지금 당장 필요한 것`과 `나중에 꼭 필요한 것`을 계속 분리해야 했다.

### Feature

- object storage, customer confirmation, audit, tenantization 이 서로 연결돼 있어 한 번에 다루기 어려웠다.
- 제품적으로는 단순해 보여도 기술적으로는 경계가 복잡했다.

### Builder

- 큰 파일에서 반복적인 route 치환 작업은 위험도가 높았다.
- 인코딩/문자열/스크립트 교체 과정에서 작은 문법 이슈가 몇 번 생겼다.
- apply_patch 대신 우회 편집이 필요했던 점은 작업 안정성을 조금 낮췄다.

### QA

- SQLite 와 Postgres 가 공존하는 과도기라, "무엇을 어디까지 테스트해야 충분한가" 판단이 계속 필요했다.
- 아직 live multitenant 시나리오까지는 검증 범위가 가지 못했다.

## Keep

- 문서 먼저, 구현 나중 원칙
- PM 주도의 batch boundary 결정
- staging preflight + smoke check 루틴
- 작은 통합 후 바로 회귀 테스트 실행

## Fix

- route 대형 파일 치환 작업은 더 작은 helper 분리와 리팩터링이 필요하다.
- owner-token / session 경로 공존에 대한 명시적 운영 정책이 더 필요하다.
- Postgres timeline write 와 object storage 연동은 늦지 않게 별도 배치로 정리해야 한다.

## Drop

- 한 배치에서 너무 많은 계층을 동시에 옮기려는 시도
- 비용이 높은 인프라를 제품 검증 전에 먼저 확정하려는 접근
- runtime cutover 를 staging readiness 와 혼동하는 판단

## PM Final Reflection

- 지금 프로젝트는 속도보다 방향이 좋다.
- 이건 장점이다.
- 특히 `배포 가능`, `검증 가능`, `되돌아볼 기록이 남는 상태`까지 왔다는 점이 중요하다.
- 다음부터는 더 화려한 기능보다, 남은 통합 리스크를 하나씩 닫는 쪽이 더 가치가 크다.
