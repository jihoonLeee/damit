# POST_LOGIN_EXCEPTION_POLISH_PLAN

## Why this batch

- 로그인 자체는 동작하지만, 예외 흐름에서 사용자가 `왜 이 화면에 왔는지`와 `다음에 무엇을 눌러야 하는지`를 바로 이해하지 못할 여지가 남아 있다.
- 특히 `세션 만료 후 복귀`, `OWNER 전용 화면 접근 제한`, `회사 전환 직후 컨텍스트 변경`은 기능적으로는 맞아도 UX 상으로는 아직 설명이 부족하다.
- 운영형 제품으로 보이려면 정상 흐름뿐 아니라 예외 흐름도 같은 톤과 구조로 정리되어야 한다.

## PM goal

- 로그인 이후 10초 안에 아래 3가지를 이해할 수 있게 만든다.
  - 왜 이 화면으로 돌아왔는지
  - 지금 활성화된 회사와 역할이 무엇인지
  - 다음에 어느 화면으로 가면 되는지

## Scope

### 1. Login exception guidance

- 로그인 화면에서 `next` 경로를 읽고, 로그인 후 돌아갈 목적지를 문장으로 보여준다.
- `session-expired`, `logged-out` 같은 복귀 이유를 목적지와 연결해서 설명한다.
- 링크 요청 전/후 안내 문구도 목적지 기준으로 더 구체화한다.

### 2. Home exception guidance

- 운영 홈 상단에서 복귀 이유를 배너로 명확히 보여준다.
- `owner-required`일 때는 막힘만 보여주지 않고, 바로 갈 수 있는 권장 화면(`/app`)을 함께 제안한다.
- 회사 전환 직후에는 현재 세션에 무엇이 바뀌었는지와, 열린 탭을 어떻게 다뤄야 하는지 안내한다.

### 3. Route recommendation clarity

- 운영 홈에서 현재 역할 기준 추천 경로를 요약 문장으로 보여준다.
- 작업 워크스페이스와 운영 콘솔의 역할 차이를 더 짧고 분명하게 정리한다.
- 회사가 없거나 컨텍스트가 비어 있을 때는 이동보다 연결 완료가 먼저라는 점을 분명히 한다.

## Non-goals

- 실제 메일 로그인 cutover
- auth/session API 구조 변경
- 회사/권한 모델 변경

## Success criteria

- 로그인 화면만 봐도 로그인 후 돌아갈 목적지를 추측하지 않아도 된다.
- 운영 홈에서 `왜 돌아왔는지`, `현재 컨텍스트`, `권장 다음 액션`이 같은 시야 안에 보인다.
- 회사 전환 직후 사용자가 `이제 무엇이 바뀌었는지`를 다시 해석하지 않아도 된다.
