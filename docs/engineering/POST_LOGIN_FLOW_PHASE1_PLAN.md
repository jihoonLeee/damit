# POST_LOGIN_FLOW_PHASE1_PLAN

## Why this batch

- `/app` 작업 흐름은 많이 정리됐지만, `로그인 -> 운영 홈 -> 작업 워크스페이스` 연결은 아직 한 번에 읽히지 않는다.
- 운영 홈에서 `지금 내가 어떤 권한으로 어디에 들어와 있는지`, `다음에 무엇을 해야 하는지`, `어디로 이동해야 하는지`가 더 직접적으로 보여야 한다.
- 실제 메일 로그인 cutover는 도메인 검증이 끝나기 전까지 HOLD이므로, 현재 상태를 제품 안에서 더 명확히 설명해야 한다.

## PM goal

- 로그인 직후 사용자가 10초 안에 아래 3가지를 이해하게 만든다.
  - 지금 로그인은 정상적으로 끝났는지
  - 현재 어떤 회사와 권한으로 들어와 있는지
  - 다음으로 `/app`과 `/ops` 중 어디로 가야 하는지

## Scope

### 1. Login screen

- 로그인 링크 요청 후 `무엇을 기다리는 상태인지`를 단계형으로 보여준다.
- 첫 로그인 setup이 필요한 경우, `왜 추가 입력이 필요한지`를 에러처럼 보이지 않게 설명한다.
- 파일 미리보기 메일 모드와 실제 메일 운영 모드를 더 명확히 구분한다.

### 2. Home screen

- 현재 세션과 회사 컨텍스트를 요약하는 운영 카드 추가
- 역할별 `다음 할 일` 카드 추가
- `/app`과 `/ops`의 차이를 설명하는 이동 가이드 추가
- OWNER가 아니면 운영 콘솔 CTA를 숨기되, 왜 숨겨지는지 흐름상 자연스럽게 보이게 한다

### 3. Mail cutover note

- 실메일 cutover는 여전히 HOLD
- 제품 표면에서 `지금은 파일 미리보기 모드인지`, `실메일 운영 준비가 안 된 이유가 코드가 아니라 도메인 검증인지`를 혼동 없이 보이게 유지한다

## Non-goals

- 실제 Resend 운영 cutover
- 새 auth API 추가
- 회사/세션 데이터 모델 변경

## Success criteria

- 로그인 성공 후 사용자가 다음 액션을 추측하지 않아도 된다
- 운영 홈에서 `현재 회사`, `현재 역할`, `다음 이동`이 상단에서 바로 읽힌다
- mail cutover가 HOLD인 이유가 제품 안에서도 분명하게 드러난다
