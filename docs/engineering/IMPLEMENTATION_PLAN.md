# IMPLEMENTATION PLAN

## 문서 목적

- `현장 추가금 합의 비서` MVP를 실제 구현 순서로 정리한다.
- PM이 중간마다 범위, 사용자 흐름, 증빙 가치가 흐트러지지 않았는지 점검할 수 있게 한다.
- 기준 문서:
  - `docs/product/PRD.md`
  - `docs/product/MVP_SCOPE.md`
  - `docs/ux/IA.md`
  - `docs/ux/SCREEN_FLOW.md`
  - `docs/ux/SCREEN_SPECS.md`
  - `docs/specs/FEATURE_SPECS.md`
  - `docs/specs/API_SPEC.md`
  - `docs/specs/DOMAIN_MODEL.md`
  - `docs/specs/ERROR_POLICY.md`

## 구현 목표

- 현장에서 20초 안에 `빠른 현장 기록`을 남길 수 있어야 한다.
- 작업 건 상세에서 `기본 포함 범위 vs 추가 작업` 대비와 변경 금액이 한 흐름으로 보여야 한다.
- 고객 설명 초안과 합의 기록이 분리되지 않고 같은 작업 건 컨텍스트 안에서 이어져야 한다.
- 모든 실패 케이스는 문서화된 에러 코드와 복구 동작으로 처리해야 한다.

## MVP 범위 재확인

### 포함

- 빠른 현장 기록 단일 생성
- 작업 건 생성
- 기존 작업 건 연결
- 작업 건 목록/상세 조회
- 변경 견적 저장
- 범위 대비 조회
- 설명 초안 생성/조회
- 합의 기록 생성
- 타임라인 조회

### 제외

- 다중 역할 권한
- 고객 서명 또는 법적 계약 기능
- 자동 결제 연동
- 자유 형식 초안 편집기
- 다업종 확장 템플릿
- 모바일 앱 패키징

## 구현 원칙

- 구현 순서는 `증빙 확보 -> 작업 건 연결 -> 금액/범위 정리 -> 설명 -> 합의 기록 -> 조회 안정화`를 따른다.
- 상태 변경의 단일 쓰기 경로는 `agreement_record` 생성이다.
- 에러 코드는 프론트/백엔드 공통 계약으로 취급한다.
- PM 검수 없이 P0 범위를 추가하지 않는다.
- UI는 프로토타입과 스펙 사이에서 `더 예쁘게`보다 `더 빠르게 설명 가능한가`를 우선한다.

## 단계별 구현 계획

### Phase 0. 착수 정렬

목표
- 스펙, 데이터 모델, PM 검수 기준을 고정한다.

작업
- 엔티티/enum/상태 전이 재확인
- API 라우트와 에러 코드 카탈로그 동결
- 구현 순서와 QA 진입 조건 확정

산출물
- 구현 브랜치 시작 기준 문서 세트
- 개발/QA 공용 체크리스트 초안

PM 체크포인트
- `빠른 현장 기록`이 반드시 사진 포함 단일 저장인지 확인
- `합의 기록`이 상태 변경 source of truth인지 확인
- MVP 제외 범위가 슬며시 다시 들어오지 않았는지 확인

### Phase 1. 도메인/백엔드 골격

목표
- 이후 기능이 기대는 데이터 모델과 공통 실패 계약을 먼저 안정화한다.

작업
- DB 스키마 및 migration 설계
- enum, 관계, 인덱스 정의
- 공통 에러 응답 포맷 구현
- requestId/logging/error mapper 구현
- OWNER 인증/인가 골격 구현

완료 기준
- DB와 API가 문서의 enum, status, required 필드와 일치한다.
- 공통 에러 응답이 모든 write endpoint에서 동일 형식으로 반환된다.

PM 체크포인트
- `job_case.current_status`를 독립 수정값이 아니라 파생/캐시 필드로 봐도 되는지 확인
- 상세 화면에 필요한 데이터가 현재 모델로 충분한지 확인

### Phase 2. 빠른 현장 기록 + 작업 건 연결

목표
- 가장 핵심인 현장 첫 입력 흐름을 끝까지 작동시킨다.

작업
- `POST /field-records` multipart 처리
- 사진 저장/검증/실패 복구
- 작업 건 생성 API
- 기존 작업 건 검색/연결 API
- `FIELD_RECORD_LINKED` 타임라인 생성
- 목록 첫 진입용 최소 화면/폼 구현

완료 기준
- 사진 1장 이상, 사유, 메모 규칙이 실제로 검증된다.
- 사진 없는 유령 레코드가 생성되지 않는다.
- 저장된 현장 기록이 새 작업 건 또는 기존 작업 건으로 안정적으로 연결된다.

PM 체크포인트
- 사진 먼저 찍고 나중에 작업 건을 보완하는 흐름이 끊기지 않는지 확인
- 현장 사용성 기준으로 입력 단계가 과하지 않은지 확인

### Phase 3. 견적/범위 대비/설명 초안

목표
- `왜 추가금이 생겼는지`를 설명 가능한 상태로 만든다.

작업
- 변경 견적 저장 API 및 차액 계산
- `scope_comparison` 생성/조회
- 설명 초안 생성 로직
- 최신 초안 1개 유지 + 타임라인 기록
- 상세 화면의 범위 대비/초안 영역 구현

완료 기준
- 원래 견적이 없으면 변경 견적 저장이 막힌다.
- 현장 기록과 변경 금액이 없으면 초안 생성이 막힌다.
- 초안 문구 구조가 스펙 순서를 따른다.

PM 체크포인트
- 화면에서 `기본 포함 범위`와 `추가 작업`의 대비가 실제로 눈에 들어오는지 확인
- 초안이 고객 설명용으로 충분히 구체적인지 확인

### Phase 4. 합의 기록/목록/상세/타임라인

목표
- 제품의 핵심 가치인 `설명 이후 무엇이 합의됐는지`를 남긴다.

작업
- 합의 기록 생성 API
- 상태 파생 갱신 로직
- 목록 상태 필터 구현
- 상세 화면의 최신 합의 기록/타임라인 구성
- `작업 제외`와 `보류` 조회 흐름 구현

완료 기준
- `AGREED`에서 확정 금액 누락 시 저장되지 않는다.
- 합의 기록 저장 후 목록/상세 상태가 일관되게 반영된다.
- 타임라인에서 연결, 초안, 합의 기록 이벤트가 순서대로 보인다.

PM 체크포인트
- 상태만 바뀌고 실제 합의 정보가 비는 경우가 없는지 확인
- `작업 제외`가 숨는 상태가 아니라 실제 회고 가능한 데이터로 남는지 확인

### Phase 5. 안정화/QA/출시 준비

목표
- P0 흐름을 끊는 버그와 문서-구현 불일치를 제거한다.

작업
- API contract test 정리
- 주요 사용자 시나리오 E2E 또는 통합 테스트
- 에러 메시지/UI 복구 점검
- 운영 로그/모니터링/수동 점검 절차 정리
- 파일럿 배포 전 smoke test 수행

완료 기준
- P0 시나리오 전부 통과
- 엔드포인트별 에러 케이스가 최소 1회 이상 검증됨
- PM/QA가 파일럿 전 출시 보류 사유가 없다고 판단

PM 체크포인트
- 첫 파일럿 사용자가 실제로 `추가금 설명 -> 합의 기록`까지 완주할 수 있는지 최종 점검
- 문구와 상태명이 현장 언어와 어긋나지 않는지 점검

## 추천 구현 순서

1. Domain model + migration
2. 공통 error contract + auth skeleton
3. 빠른 현장 기록 생성
4. 작업 건 생성/검색/연결
5. 작업 건 상세 최소 조회
6. 변경 견적 저장
7. 범위 대비 조회
8. 설명 초안 생성/조회
9. 합의 기록 생성
10. 목록 필터/타임라인
11. 회귀 테스트/운영 점검

## PM 게이트

| Gate | 시점 | PM 질문 | 통과 기준 |
|---|---|---|---|
| `G0` | 개발 시작 전 | MVP 범위가 정말 잠겼는가 | 제외 범위 재유입 없음 |
| `G1` | Phase 1 후 | 데이터 모델이 UX를 버틸 수 있는가 | 합의 내용, 범위 대비, 타임라인 표현 가능 |
| `G2` | Phase 2 후 | 현장 20초 입력이 가능한가 | 사진 우선 흐름 작동 |
| `G3` | Phase 3 후 | 고객 설명 가치가 실제로 느껴지는가 | 범위 대비 + 초안 품질 통과 |
| `G4` | Phase 4 후 | 상태와 합의 내용이 일치하는가 | 합의 기록과 현재 상태 일관 |
| `G5` | 출시 직전 | 파일럿에 내보내도 되는가 | P0 결함 없음 |

## 주요 리스크와 대응

| 리스크 | 설명 | 대응 |
|---|---|---|
| 현장 입력 피로 | 기록 단계가 길어지면 사용하지 않음 | 사진 우선, 필수 입력 최소화, PM 현장 사용성 점검 |
| 사유 카테고리 과잉/부족 | 카테고리가 현실과 안 맞으면 설명 초안 품질 저하 | 초기엔 입주청소 한 업종으로 고정, OTHER 메모 허용 |
| 상태-합의 불일치 | 상태는 바뀌었는데 합의 내용이 비어있을 수 있음 | 상태 변경 경로를 agreement_record 단일화 |
| 에러 계약 불일치 | 프론트/백이 다른 실패 메시지를 처리할 수 있음 | 공통 에러 코드 테스트 필수 |
| 범위 확장 유혹 | 파일럿 전에 권한/자동화/다업종이 끼어들 수 있음 | PM 게이트에서 차단 |

## Handoff

### Builder에게

- Phase 단위로 PR 또는 작업 묶음을 끊는다.
- 각 Phase 종료 시 PM 게이트 체크 결과를 남긴다.
- 스펙과 다른 판단이 필요하면 먼저 `DECISIONS.md` 후보로 올린다.

### QA에게

- `P0 사용자 시나리오`, `에러 코드`, `상태 전이`, `타임라인 생성`을 핵심 회귀 축으로 본다.
- 특히 `사진 누락`, `이미 연결된 현장 기록`, `AGREED 금액 누락`, `잘못된 상태 필터`를 반복 확인한다.
