# PM + Move-In Cleaning Expert Review

## 문서 목적

- `현장 추가금 합의 비서`의 UX 문서를 PM 관점과 입주청소 현장 전문가 관점에서 함께 검증한다.
- 이번 리뷰는 `예뻐 보이는 UX`보다 `현장에서 실제로 말싸움을 줄이고 추가금 설명을 방어할 수 있는가`를 기준으로 본다.

## 참여자

- `pm-brainstorm`
- `move-in-cleaning-expert`

## 한 줄 결론

- 방향은 좋지만, 아직 `합의가 무엇으로 증명되는가`, `기본 포함 범위와 추가 범위가 어떻게 대비되는가`, `현장에서 가장 먼저 무엇을 기록하는가`가 UX에 충분히 드러나지 않는다.

## Findings

### 1. P1 - 합의 상태는 있는데 합의 내용이 없다

- 현재 문서는 `설명완료`, `합의완료`, `보류`, `작업 제외` 상태를 남길 수 있지만, 실제로 고객이 `어떤 금액`, `어떤 범위`, `어떤 채널`로 확인했는지는 남기지 않는다.
- PM 관점에서는 제품의 핵심 가치인 `합의 초안 + 증빙 기록`이 약해지고, 현장 관점에서는 나중에 고객이 `그 금액 얘기 못 들었다`고 말했을 때 방어력이 부족하다.
- 최소한 아래 정보는 P0 수준으로 검토해야 한다.
- 제안:
  - 확인 채널
  - 확인 시각
  - 확정 금액
  - 고객 반응 메모
- 참고:
  - `docs/ux/IA.md:52`
  - `docs/ux/SCREEN_FLOW.md:70`
  - `docs/ux/SCREEN_SPECS.md:207`

### 2. P1 - 기본 청소 범위와 추가 범위의 대비가 UX에 없다

- 현재는 `추가 사유`, `변경 금액`, `설명 초안`은 있지만 고객이 가장 많이 묻는 질문인 `원래 포함 아닌가요?`에 대응하는 구조가 없다.
- 입주청소에서는 추가금 설득의 핵심이 `지금 상태가 나쁘다`가 아니라 `이건 기본 범위 밖이다` 또는 `사전 가정과 실제 상태가 다르다`를 보여주는 데 있다.
- 제안:
  - 작업 건 상세나 설명 초안 생성 영역에서 `기본 포함 범위`와 `추가 작업`의 대비를 보여준다.
  - 설명 초안도 `현장 상태 -> 추가 작업 -> 왜 기본 범위 밖인지 -> 금액 변화` 순서를 기본으로 둔다.
- 참고:
  - `docs/ux/IA.md:64`
  - `docs/ux/SCREEN_SPECS.md:167`
  - `docs/product/MVP_SCOPE.md:81`

### 3. P1 - 현장 첫 입력 흐름이 너무 관리형이다

- 현재 UX는 `작업 건 생성 -> 고객명/주소/원래 견적 입력 -> 현장 이슈 입력` 순서에 가깝다.
- 하지만 실제 입주청소 현장에서는 문제가 보이면 가장 먼저 하는 행동은 `사진 확보`다. 기본 정보가 늦어도 사진이 먼저 남아야 한다.
- 이 흐름은 파일럿에서 `귀찮아서 안 씀`으로 이어질 가능성이 있다.
- 제안:
  - `빠른 현장 기록` 진입점을 두고 사진부터 찍게 한다.
  - 고객명/주소/원래 견적은 사진 저장 후 보완하도록 순서를 재배치한다.
- 참고:
  - `docs/ux/SCREEN_FLOW.md:18`
  - `docs/ux/SCREEN_SPECS.md:56`
  - `docs/ux/IA.md:177`

### 4. P2 - 추가 사유 카테고리가 아직 고객 반발을 버틸 만큼 구체적이지 않다

- `심한 오염`, `범위 추가`, `기타` 같은 상위 표현은 내부 기록에는 도움이 되지만 고객 설명이나 현장 선택에는 다소 넓다.
- 입주청소 사장님은 `니코틴`, `곰팡이`, `스티커 제거`, `폐기물`, `베란다 추가`, `다용도실`, `창고`처럼 더 손에 잡히는 이유로 말한다.
- 제안:
  - 카테고리를 1차와 2차 구조로 나누거나, 상위 카테고리 아래 대표 예시를 함께 노출한다.
  - 설명 초안도 상위 카테고리 그대로 쓰지 말고 고객이 이해할 구체 표현으로 바꾼다.
- 참고:
  - `docs/ux/SCREEN_SPECS.md:148`
  - `docs/product/MVP_SCOPE.md:81`

### 5. P2 - `작업 제외`와 협의 실패 데이터가 학습 구조에 약하다

- 문서에는 `작업 제외` 상태가 있지만 목록 필터나 운영 해석에서는 비중이 낮다.
- 그런데 실제 입주청소에서는 `추가 작업을 포기한 이유`, `설명했지만 거절된 이유`가 다음 세일즈와 템플릿 개선에 매우 중요하다.
- 제안:
  - 목록 필터에 `작업 제외`를 넣거나, 최소한 상세 타임라인에서 이유 메모를 강조한다.
  - 향후 파일럿 분석 항목으로 `거절 사유`를 수집한다.
- 참고:
  - `docs/ux/IA.md:152`
  - `docs/ux/IA.md:200`
  - `docs/ux/SCREEN_SPECS.md:215`

### 6. P3 - MVP 에서 설정 탭은 비중이 크고 현장 가치와 거리가 있다

- 현재 IA 는 하단 탭 두 개 중 하나를 `설정`에 쓴다.
- 하지만 첫 파일럿에서 반복적으로 열릴 가능성이 높은 것은 `작업 건`뿐이고, 설정은 운영자용 보조 수준에 가깝다.
- 제안:
  - 설정을 별도 탭에서 빼고 숨은 보조 화면이나 프로필 진입으로 내린다.
- 참고:
  - `docs/ux/IA.md:69`
  - `docs/ux/IA.md:117`

## PM + 전문가 대화 요약

> PM: "방향은 맞는데, 아직 상태값 중심이라 합의 증빙 제품처럼 느껴지진 않는다."
>
> 입주청소 전문가: "현장에서는 사진보다도 '왜 기본 청소 밖인지' 설명이 더 중요하다. 그 대비가 화면에 더 나와야 한다."
>
> PM: "첫 입력도 너무 차분한 관리형이다. 현장에선 사진부터 잡아야 한다."
>
> 입주청소 전문가: "맞다. 고객명은 나중에 적어도 되지만 현장 사진은 그 순간 아니면 놓친다."

## 유지해도 좋은 점

- `법률 계약 도구`가 아니라 `설명/기록 도구`로 경계를 잡은 점은 적절하다.
- `입주청소` 한 업종으로 좁힌 점은 좋다.
- `설명 초안`과 `상태 기록`을 P0 으로 둔 방향은 맞다.
