# SCREEN FLOW

## 문서 목적

- `현장 추가금 합의 비서` MVP의 주요 사용자 흐름을 정의한다.
- 첫 버전은 입주청소 사장님이 현장에서 `사진 먼저 확보하고`, `추가 사유를 구체화하고`, `고객 설명과 합의 내용까지 남기는 흐름`만 다룬다.

## 핵심 플로우 한 줄 요약

- `빠른 현장 기록 -> 작업 건 연결/보완 -> 변경 견적 입력 -> 고객 설명 초안 확인 -> 합의 내용 기록`

## Flow 1. 빠른 현장 기록 시작

### 시작 조건

- 사용자가 현장에서 예상과 다른 상태를 발견했다.

### 흐름

1. 사용자는 `작업 건 목록` 또는 첫 화면에서 `빠른 현장 기록` CTA를 누른다.
2. 즉시 카메라 또는 사진 업로드 화면이 열린다.
3. 사용자는 사진을 1장 이상 올린다.
4. 1차 사유 카테고리를 선택한다.
5. 2차 상세 사유를 선택하거나 짧은 메모를 남긴다.
6. 시스템은 현장 기록을 임시 저장한다.
7. 사용자는 기존 작업 건에 연결하거나 새 작업 건을 만든다.

### 성공 조건

- 고객명/주소를 몰라도 현장 기록을 먼저 남길 수 있어야 한다.
- 사진 저장까지 20초 안에 끝나는 것이 이상적이다.

## Flow 2. 작업 건 생성 또는 보완

### 시작 조건

- 빠른 현장 기록이 저장되었지만 작업 건 정보가 충분하지 않다.

### 흐름

1. 사용자는 `새 작업 건 만들기` 또는 `기존 작업 건 연결` 중 하나를 선택한다.
2. 새 작업 건일 경우 고객명 또는 현장 식별명, 주소, 원래 견적 금액을 입력한다.
3. 저장 후 자동으로 `작업 건 상세`로 이동한다.
4. 방금 남긴 현장 기록이 상세에 연결되어 보인다.

### 성공 조건

- 빠른 기록에서 상세까지 흐름이 끊기지 않아야 한다.
- 작업 건 보완은 현장 기록 이후에 하도록 순서가 유지되어야 한다.

## Flow 3. 범위 대비와 변경 견적 정리

### 시작 조건

- 작업 건에 현장 기록이 1건 이상 연결되어 있다.

### 흐름

1. 사용자는 상세 화면에서 `기본 포함 범위 vs 추가 작업` 카드를 본다.
2. 시스템은 선택한 사유를 바탕으로 기본 포함 범위 요약과 추가 작업 요약을 보여준다.
3. 사용자는 원래 견적 금액을 확인한다.
4. 변경 후 금액을 입력한다.
5. 시스템은 차액을 계산한다.

### 성공 조건

- 고객이 자주 묻는 `원래 포함 아닌가요?`에 답할 근거가 화면에 있어야 한다.
- 차액은 수동 계산 없이 바로 보여야 한다.

## Flow 4. 고객 설명 초안 생성

### 시작 조건

- 현장 기록과 금액 변경 정보가 있다.

### 흐름

1. 사용자는 `설명 초안 생성`을 누른다.
2. 시스템은 아래 순서의 고객용 설명 문구를 만든다.
3. 순서:
   - 현장 상태 설명
   - 추가 작업 항목
   - 왜 기본 범위 밖인지 또는 왜 사전 가정과 다른지
   - 금액 변경 내용
   - 진행 여부 확인 요청
4. 사용자는 문구를 복사하거나 바로 읽고 고객에게 설명한다.

### 성공 조건

- 초안은 10초 이내 보여야 한다.
- 초안은 메신저에 바로 붙여 넣을 수 있는 길이여야 한다.
- 상위 카테고리 문구가 아니라 고객이 이해할 구체 표현이어야 한다.

## Flow 5. 합의 내용 기록

### 시작 조건

- 고객에게 설명을 마쳤거나 답변을 받았다.

### 흐름

1. 사용자는 `합의 기록` 영역을 연다.
2. 상태값을 선택한다.
3. 상태값 예시:
   - 설명완료
   - 합의완료
   - 보류
   - 작업 제외
4. 사용자는 확인 채널을 고른다.
5. 확인 시각과 확정 금액을 남긴다.
6. 필요하면 고객 반응 메모를 남긴다.
7. 저장하면 상세 상단과 목록 카드 상태가 갱신된다.

### 성공 조건

- 상태만 바꾸는 것이 아니라 실제 합의 내용을 남겨야 한다.
- 합의완료일 때는 확정 금액이 함께 남는 것이 기본이다.

## Flow 6. 나중에 분쟁 또는 이의가 생겼을 때 재조회

### 시작 조건

- 같은 작업 건을 다시 확인해야 한다.

### 흐름

1. 사용자는 작업 건 목록에서 해당 건을 찾는다.
2. 상태 필터 또는 최근 업데이트 기준으로 진입한다.
3. 상세 화면에서 사진, 상세 사유, 기본 범위 대비, 변경 금액, 설명 초안, 합의 기록을 확인한다.
4. 필요하면 상태를 다시 바꾸거나 보완 메모를 남긴다.

### 성공 조건

- 상세 화면에서 핵심 근거를 스크롤 1~2번 안에 확인할 수 있어야 한다.
- `무엇을 설명했고 무엇에 동의했는지`가 한눈에 보여야 한다.

## 보조 플로우

### 보조 Flow A. 설명 없이 보류 처리

1. 현장 기록은 남겼지만 고객과 즉시 통화가 어렵다.
2. 사용자는 설명 초안만 생성해둔다.
3. 상태를 `보류`로 저장한다.
4. 목록에서 보류 필터로 다시 확인한다.

### 보조 Flow B. 추가 작업 거절 또는 작업 제외

1. 고객이 추가 작업을 원하지 않거나 협의가 결렬된다.
2. 사용자는 상태를 `작업 제외`로 저장한다.
3. 거절 또는 제외 이유를 메모로 남긴다.
4. 해당 이력은 타임라인에 남고 목록 필터에서도 찾을 수 있다.

## 에지 케이스

### 사진 없이 진행하려는 경우

- 첫 MVP는 `사진 1장 필수`로 고정한다.
- 이유: 이 제품의 핵심 가치가 `설명 + 증빙`이기 때문이다.

### 금액을 아직 모르는 경우

- 현장 기록은 먼저 저장 가능하다.
- 변경 견적 입력은 나중에 해도 된다.
- 다만 설명 초안 생성 전에는 `금액 미입력` 상태를 분명히 알려준다.

### 고객명 또는 주소를 나중에 아는 경우

- 빠른 현장 기록은 임시 상태로 저장한다.
- 작업 건 보완 단계에서 고객명, 주소, 원래 견적을 나중에 채운다.

### 초안이 마음에 들지 않는 경우

- 초안은 복사 후 직접 수정해서 쓰는 전제를 둔다.
- MVP에서는 자유 편집기보다 `다시 생성` 또는 `메모 보완` 정도만 제공한다.

## 화면 간 전환 원칙

- 목록에서 빠른 기록으로 가는 CTA 는 가장 눈에 띄어야 한다.
- 입력과 결과는 가능한 한 같은 상세 페이지 안에서 이어져야 한다.
- 합의 기록은 설명 초안과 멀리 떨어뜨리지 않는다.
- 모달/바텀시트는 짧은 사유 선택, 상태 선택 정도에만 쓴다.

## 플로우 우선순위

### P0

- Flow 1 빠른 현장 기록 시작
- Flow 2 작업 건 생성 또는 보완
- Flow 3 범위 대비와 변경 견적 정리
- Flow 4 고객 설명 초안 생성
- Flow 5 합의 내용 기록

### P1

- Flow 6 재조회
- 보조 Flow A 보류 처리
- 보조 Flow B 작업 제외 처리

## 플로우 설계 결론

- MVP의 본질은 `새 작업 건을 예쁘게 관리하는 것`이 아니라 `문제가 보인 순간 증빙을 남기고, 설명하고, 합의 내용까지 기록하는 것`이다.
- 따라서 모든 플로우는 `빠른 현장 기록`을 시작점으로 하고, 상세 화면에서 설명과 합의가 자연스럽게 이어지도록 설계한다.
