# IA

## 문서 목적

- `현장 추가금 합의 비서` MVP의 정보 구조를 정의한다.
- 입주청소 현장에서 `문제 발견 -> 사진 확보 -> 추가 사유 정리 -> 범위 대비 확인 -> 설명 초안 확인 -> 합의 내용 기록`까지 가장 짧게 끝내도록 설계한다.
- 첫 버전은 `입주청소` 업종에만 맞춘다.

## UX 목표

- 현장에서 가장 먼저 해야 할 행동은 `사진 확보`여야 한다.
- 사용자는 고객 기본정보를 다 적지 않아도 현장 기록을 먼저 남길 수 있어야 한다.
- 고객에게 보여줄 설명 초안뿐 아니라 `무엇에 동의했는지`가 함께 남아야 한다.
- 제품은 `관리 시스템`보다 `현장 합의/분쟁 방지 도구`처럼 느껴져야 한다.

## 핵심 사용자

- 입주청소 사장님
- 1인 또는 2~3인 팀 리더
- 현장에서 고객과 직접 설명하고 합의를 끝내는 사람

## 핵심 객체

### 1. 작업 건

- 현장 단위의 최상위 컨테이너
- 포함 정보:
  - 고객명 또는 현장 식별명
  - 연락처 메모
  - 주소 또는 현장명
  - 원래 견적 금액
  - 현재 합의 상태
  - 최근 업데이트 시각

### 2. 현장 기록

- 현장에서 발견한 문제의 최초 증빙
- 포함 정보:
  - 사진
  - 1차 사유 카테고리
  - 2차 상세 사유 또는 예시 선택
  - 짧은 메모
  - 기록 시각

### 3. 범위 대비

- 기본 청소 범위와 추가 작업 범위를 비교해 보여주는 객체
- 포함 정보:
  - 기본 포함 범위 요약
  - 현장 추가 작업 요약
  - 왜 기본 범위 밖인지 설명 포인트

### 4. 변경 견적

- 금액 변화와 고객 설명 문맥을 담는 객체
- 포함 정보:
  - 원래 견적 금액
  - 변경 후 금액
  - 차액
  - 고객 설명 초안

### 5. 합의 기록

- 고객이 무엇을 어떤 채널로 확인했는지 남기는 객체
- 포함 정보:
  - 합의 상태
  - 확인 채널
  - 확인 시각
  - 확정 금액
  - 고객 반응 메모

## 정보 구조 원칙

- `작업 건`이 최상위 단위다.
- 현장에서는 `빠른 현장 기록`이 작업 건 생성보다 앞설 수 있다.
- 한 작업 건 안에 `현장 기록`, `범위 대비`, `변경 견적`, `합의 기록`, `타임라인`이 묶여 있다.
- 상세 화면에서는 `고객 설명`과 `합의 내용`이 함께 보여야 한다.
- 메뉴 깊이는 최대 2단계로 제한한다.

## 글로벌 내비게이션

### 기본 구조

- 하단 탭 1개
- 탭:
  - `작업 건`
- 보조 진입:
  - 우측 상단 `더보기` 또는 프로필 아이콘

### 이 구조를 택한 이유

- MVP에서 사용자가 반복적으로 드나드는 영역은 사실상 `작업 건` 하나다.
- 설정은 현장 행동을 방해하지 않도록 보조 화면으로 내린다.

## 화면 트리

```text
작업 건
├─ 작업 건 목록
│  ├─ 필터: 전체 / 미설명 / 설명완료 / 합의완료 / 보류 / 작업 제외
│  ├─ 빠른 현장 기록 CTA
│  └─ 작업 건 카드 선택
├─ 빠른 현장 기록
│  ├─ 사진 업로드
│  ├─ 1차 사유 선택
│  ├─ 2차 상세 사유 선택
│  └─ 임시 저장 후 작업 건 연결
├─ 작업 건 생성/보완
└─ 작업 건 상세
   ├─ 요약 헤더
   ├─ 고객 설명 초안
   ├─ 합의 기록 카드
   ├─ 범위 대비 카드
   ├─ 현장 기록 카드
   ├─ 변경 견적 입력
   └─ 타임라인

보조 화면
└─ 설정
   ├─ 업종 정보 보기
   ├─ 기본 포함 범위 안내 보기
   └─ 문구 톤 가이드 보기
```

## 핵심 화면 우선순위

### P0

- 작업 건 목록
- 빠른 현장 기록
- 작업 건 생성/보완
- 작업 건 상세
- 변경 견적/고객 설명 초안
- 합의 기록 입력

### P1

- 설정
- 기본 포함 범위 안내

## 작업 건 상세 내부 구조

### 상단 요약 영역

- 고객명 또는 현장명
- 주소
- 현재 합의 상태 배지
- 원래 견적 금액
- 변경 후 금액
- 차액

### 본문 우선순위

1. 고객 설명 초안
2. 합의 기록 카드
3. 기본 포함 범위 vs 추가 작업 카드
4. 현장 사진 및 사유
5. 타임라인

### 이 순서를 택한 이유

- 사용자가 상세에 들어오는 주된 이유는 `지금 어떻게 설명할지`와 `무엇이 합의됐는지`를 보기 위해서다.
- 사진은 중요하지만 설명과 합의 문맥 아래에 묶여야 분쟁 방어력이 생긴다.

## 상태 구조

### 작업 건 상태

- 미설명
- 설명완료
- 합의완료
- 보류
- 작업 제외

### 상태 의미

- `미설명`: 현장 기록은 남았지만 고객 설명 전
- `설명완료`: 설명은 했지만 확정 답변을 못 받았거나 추가 확인 필요
- `합의완료`: 고객이 변경 금액과 작업 범위에 동의
- `보류`: 고객 회신 대기 또는 현장 판단 보류
- `작업 제외`: 추가 작업을 진행하지 않기로 결정

### 상태 전이

- 미설명 -> 설명완료
- 설명완료 -> 합의완료
- 설명완료 -> 보류
- 설명완료 -> 작업 제외
- 보류 -> 설명완료 또는 합의완료 또는 작업 제외

## 빈 상태 설계 원칙

### 목록 빈 상태

- `빠른 현장 기록` CTA를 가장 크게 보여준다.
- 보조 CTA 로 `작업 건 직접 만들기`를 둔다.
- 문구는 `문제가 보이면 먼저 사진부터 남겨보세요`처럼 현장 행동을 유도한다.

### 상세 빈 상태

- 현장 기록이 없으면 `사진과 사유를 먼저 남겨주세요` 안내 카드가 가장 먼저 보인다.
- 합의 기록이 없으면 `설명 후 고객 반응을 남겨두세요` 안내가 보인다.

## 에러/예외 구조

- 사진 업로드 실패
- 상세 사유 미선택
- 금액 미입력
- 초안 생성 실패
- 합의 기록 저장 실패

### UX 원칙

- 에러는 기술 원인보다 다음 행동을 먼저 보여준다.
- 예: `사진을 다시 올려주세요`, `상세 사유를 선택해주세요`, `확정 금액을 입력해주세요`

## 확장 대비 구조

- 향후 `고객 확인 링크`, `읽음 기록`, `음성 메모`, `A/S 연동`이 들어와도 `합의 기록` 또는 `상세 화면` 아래에 붙일 수 있게 설계한다.
- 하지만 MVP IA에는 확장 메뉴를 드러내지 않는다.

## IA 결론

- MVP는 `작업 건` 중심 구조를 유지하되, 입력 시작점을 `빠른 현장 기록`으로 당겨 현장 사용성을 높인다.
- 또한 `설명 초안`과 `합의 기록`을 나란히 두어 이 제품이 단순 메모장이 아니라 `합의/증빙 도구`임을 분명히 한다.
