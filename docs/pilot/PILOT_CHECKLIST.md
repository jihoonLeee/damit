# PILOT_CHECKLIST

## 문서 목적

- `현장 추가금 합의 비서`의 첫 파일럿을 작은 범위에서 안전하게 실행한다.
- PM이 현장 반응, 사용성, 반복 의향을 한 번에 볼 수 있게 한다.
- 첫 파일럿은 `입주청소 사장님 1~2명`, `실제 현장 3~5건` 기준으로 운영한다.

## 파일럿 목표

- 사장님이 `현장 문제 발견 -> 설명 초안 -> 합의 기록`까지 완주할 수 있는지 확인한다.
- 추가금 설명이 빨라졌다고 느끼는지 확인한다.
- 기록이 귀찮지 않고 오히려 분쟁 방지에 도움이 된다고 느끼는지 확인한다.

## 성공 기준

- 테스트 사용자 2명 중 1명 이상이 `계속 써볼 의향 있음`을 말한다.
- 실제 현장 3건 이상에서 `설명 초안` 또는 `합의 기록`이 끝까지 사용된다.
- `복사 -> 카카오톡/문자 붙여넣기`가 1회 이상 실제로 문제 없이 작동한다.
- 사용자가 `왜 추가금이 생겼는지 설명하기 쉬워졌다`고 말한다.

## 실패 기준

- 현장 기록 입력이 귀찮아서 중간에 포기한다.
- 설명 초안이 실제 말투/현장 상황과 너무 어긋난다.
- 합의 기록이 남아도 사장님이 가치 없다고 느낀다.
- 빠른 이동, 복사, 금액 저장 같은 핵심 액션에서 반복적인 막힘이 나온다.

## 대상 사용자 조건

- 입주청소를 주업 또는 반주업으로 실제 수행한다.
- 추가금 설명 경험이 있고, 고객과 카카오톡/전화로 소통한다.
- 현장 사진 촬영과 모바일 브라우저 사용에 거부감이 없다.
- 가능하면 `당근 문의`, `전화 문의`, `기존 고객 소개`가 섞여 있는 사용자가 좋다.

## 파일럿 전 준비

### PM 준비

- 테스트 목표 1개만 다시 말할 수 있어야 한다.
  - `설명 속도와 합의 기록 가치가 실제로 느껴지는가`
- 관찰자는 판매 모드가 아니라 학습 모드로 들어간다.
- 인터뷰 문구는 유도 질문 없이 읽는다.

### 제품 준비

- [RUNBOOK.md](/D:/AI_CODEX_DESKTOP/docs/engineering/RUNBOOK.md) 기준 서버 실행 가능 상태 확인
- [LIVE_BROWSER_REVIEW.md](/D:/AI_CODEX_DESKTOP/docs/engineering/LIVE_BROWSER_REVIEW.md) 기준 브라우저 점검 완료
- 샘플 데이터 대신 파일럿용 신규 데이터 또는 초기화 상태 준비
- health check 확인
- 모바일 브라우저 1대 이상 준비

### 현장 준비

- 테스트 사용자에게 `실제 계약 도구가 아니라 설명/기록 보조 도구`라고 미리 안내
- 20~30분 정도 관찰 허용 가능 시간 확보
- 가능하면 실제 추가금 가능성이 있는 현장 또는 최근 사례 준비

## 파일럿 진행 순서

1. 시작 전 3분
- 평소 추가금 설명을 어떻게 하는지 먼저 듣는다.
- 지금 도구는 `설명 빠르게 하기 + 합의 남기기` 목적이라고 짧게 설명한다.

2. 첫 사용 10분
- 사용자가 직접 사진 기록부터 시작하게 둔다.
- 막히더라도 바로 설명하지 말고 5~10초 정도 관찰한다.
- 어디에서 멈췄는지, 무슨 표현을 했는지 적는다.

3. 실제 흐름 10분
- 작업 건 연결
- 변경 금액 저장
- 설명 초안 생성
- 복사 후 카카오톡/문자 붙여넣기
- 합의 기록 저장

4. 회고 10분
- 가장 편한 부분
- 가장 귀찮은 부분
- 계속 쓸지 여부
- 돈 낼 가치가 있는지 여부

## 관찰 포인트

- 사진을 먼저 찍고 싶은 본능과 화면 구조가 맞는가
- 사유 선택이 빠른가, 아니면 카테고리에서 막히는가
- `금액 저장` 버튼을 바로 인지하는가
- 설명 초안이 실제 말투와 충분히 가까운가
- 복사 후 메신저로 옮기는 흐름이 자연스러운가
- 합의 기록을 남기는 순간을 부담스럽게 느끼는가
- quick jump를 스스로 발견하고 쓰는가

## 파일럿 후 바로 기록해야 할 항목

- 현장 유형
- 문의 유입 채널
- 실제 추가금 발생 여부
- 초안 사용 여부
- 복사 기능 사용 여부
- 합의 기록 저장 여부
- 사용자가 막힌 순간 1~3개
- 사용자 직설 코멘트 1~3개

## PM 판정 프레임

### Keep

- 사용자가 스스로 반복 사용하려는 신호가 있는 기능
- 말 그대로 `시간 아꼈다`고 반응한 기능
- 분쟁 대비에 의미 있다고 반응한 기록 구조

### Fix

- 가치가 있는데 한 번에 안 보이는 기능
- 용어가 어색하거나 버튼이 늦게 보이는 부분
- 실제 현장에서는 순서가 꼬이는 흐름

### Drop

- 쓰지 않는 섹션
- 설명해도 이해 안 되는 상태값
- 파일럿에서 반복적으로 무시되는 정보

## 파일럿 종료 후 산출물

- 관찰 로그 1부
- 인터뷰 요약 1부
- Keep / Fix / Drop 정리 1부
- 다음 주 패치 우선순위 3개
