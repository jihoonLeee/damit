# SKILL STANDARDS

## 목적
이 문서는 저장소의 모든 `skills/*/SKILL.md`가 같은 운영 품질을 따르도록 만드는 기준이다.
이 저장소의 스킬은 단순 설명문이 아니라, 멀티 에이전트 워크플로우를 안정적으로 반복시키는 `운영 매뉴얼`이어야 한다.

핵심 질문은 항상 아래 8개다.
- 무엇을 한다
- 무엇을 하지 않는다
- 언제 시작해야 한다
- 시작 전에 무엇을 확인해야 한다
- 무엇을 입력으로 받는다
- 무엇을 출력으로 남긴다
- 완료 전에 무엇을 다시 점검해야 한다
- 다음 단계에 무엇을 넘겨야 한다

## 운영 원칙
- Skill 은 하나의 주 책임만 가진다.
- Skill 은 입력, 출력, 금지사항, handoff 가 반드시 있어야 한다.
- Skill 은 시작 체크와 완료 체크를 반드시 가진다.
- Skill 은 산출물뿐 아니라 `작업 기억 문서`도 남겨야 한다.
- Skill 은 다음 단계가 질문 없이 이어받을 수 있는 handoff packet 을 남겨야 한다.
- Skill 은 완료 선언 전에 self-check 또는 QA gate 를 거쳐야 한다.

## AI Implement System Alignment
이 저장소의 Skill 은 아래 네 가지 시스템을 내장해야 한다.

### 1. 자동 매뉴얼 시스템
- 각 Skill 은 작업 시작 시점에 붙는 매뉴얼이어야 한다.
- `언제 사용하나`, `시작 체크`, `작업 절차`, `완료 체크`가 분명해야 한다.

### 2. 작업 기억 시스템
- 각 Skill 은 긴 작업의 맥락을 잃지 않도록 memory artifact 를 남겨야 한다.
- 예: `DECISIONS`, `MID_PROJECT_LOG`, `REVIEW`, `QA_HANDOFF`, `CHECKLIST`

### 3. 자동 품질 검사 시스템
- 각 Skill 은 종료 전에 self-check 또는 다음 단계 gate 를 거쳐야 한다.
- Builder 와 QA 는 특히 테스트 결과와 미검증 영역을 반드시 남긴다.

### 4. 전문 에이전트 시스템
- Skill 은 자신의 역할만 수행하고, 다음 역할에 넘길 정보를 남긴다.
- PM, UX, Spec, Builder, QA, Domain, Security 간 경계가 흐려지지 않아야 한다.

## 필수 구성요소
모든 `SKILL.md`는 최소 아래 섹션을 포함한다.

1. Frontmatter
- `name`
- `description`

2. 역할
- 이 Skill 의 주 책임
- 다루는 범위
- 다루지 않는 범위

3. 언제 사용하나
- 트리거 조건
- 선행 단계 또는 선행 문서

4. 시작 체크
- 작업 전 반드시 확인할 문서
- 입력이 부족할 때의 기본 가정
- 이 Skill 을 시작해도 되는 기준

5. 입력
- 필수 입력 문서
- 선택 입력 문서
- 참고해야 할 보조 근거

6. 출력
- 산출물 파일 경로
- 산출물마다 반드시 포함할 항목
- memory artifact 가 있으면 그 경로

7. 작업 절차
- 3~7단계 정도의 반복 가능한 순서
- 필요 시 의사결정 포인트 포함

8. 판단 원칙
- 우선순위를 어떻게 정하는지
- trade-off 를 어떻게 결정하는지
- 범위를 좁히는 기준과 넓히지 않는 기준

9. 완료 체크
- 완료로 보기 위한 조건
- self-review 항목
- 흔한 실패 패턴 방지 문구

10. 금지
- 범위 확장 금지
- 근거 없는 추정 금지
- 다른 Skill 역할 침범 금지

11. Handoff
- 다음 Skill 또는 다음 단계
- 반드시 넘겨야 하는 정보 묶음
- 다음 단계가 먼저 읽을 문서

## 권장 추가 요소
- 예시 입력
- 예시 출력
- release gate 또는 stop condition
- 협업 매트릭스

## Description 규칙
설명은 아래 세 요소를 모두 포함한다.
- 언제 호출되는지
- 무엇을 생산하는지
- 어디에 반영하는지

좋은 예시
- `PRD와 UX 설계를 구현 가능한 기능/데이터/API 명세로 변환해 docs/specs 아래 문서를 작성하는 스킬`

## 입력 규칙
- 필수 입력과 선택 입력을 구분한다.
- 문서 경로는 저장소 기준으로 적는다.
- 입력이 없을 때 어떤 가정을 둘지 적는다.
- 입력이 부족하면 무리하게 결과를 단정하지 말고 범위를 줄인다.

## 출력 규칙
- 출력 파일 경로를 구체적으로 적는다.
- output 형식을 고정할 수 있으면 최소 구조를 적는다.
- 기존 문서를 갱신하는지 새 문서를 만드는지 명시한다.
- 가능하면 memory artifact 를 최소 1개 포함한다.

## 작업 기억 규칙
각 Skill 은 아래 중 최소 하나를 남긴다.
- `...REVIEW.md`
- `...HANDOFF.md`
- `...CHECKLIST.md`
- `DECISIONS.md` 갱신
- `MID_PROJECT_LOG.md` 갱신

## 품질 및 QA 규칙
- 완료 선언 전에 self-check 를 거친다.
- Builder 는 테스트 결과와 미검증 영역을 숨기지 않는다.
- QA 는 blocker, medium risk, 미검증을 분리해서 적는다.
- PM/UX/Spec 도 다음 단계가 다시 묻지 않아도 되는 handoff 가 있어야 한다.

## Handoff Packet 규칙
모든 Skill 은 다음 단계에 아래 중 필요한 것을 남긴다.
- 핵심 결정 사항
- 미결정 사항
- 리스크
- 제외한 항목
- 다음 단계가 먼저 확인해야 할 문서
- 완료 기준 또는 stop condition

## 2차 검증 기준
2차 검증을 통과한 Skill 은 아래를 만족해야 한다.
- 입력/출력/금지/handoff 가 모두 있다.
- 시작 체크와 완료 체크가 있다.
- 작업 기억 문서가 정의돼 있다.
- 의사결정 원칙이 적혀 있다.
- 다음 단계가 바로 실행 가능한 handoff packet 이 있다.
- 역할이 다른 Skill 과 명확히 구분된다.
