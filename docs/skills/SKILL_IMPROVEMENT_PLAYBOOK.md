# SKILL IMPROVEMENT PLAYBOOK

## 목적
기존 Skill 을 더 구체적이고, 더 안전하고, 더 운영 가능하게 개선하기 위한 실전 가이드다.
이 문서는 Skill 리뷰와 리팩토링을 할 때 공통 절차로 사용한다.

## 언제 이 플레이북을 쓰나
- Skill 이 너무 짧어서 실제 작업 지시로 쓰기 어려울 때
- 다른 Skill 과 역할이 겹칠 때
- 입력/출력/금지/handoff 가 빠져 있을 때
- 다음 단계에서 계속 재질문이 발생할 때

## 개선 절차
1. 현재 Skill 의 한 줄 요약을 만든다.
2. 주 책임과 비책임을 분리한다.
3. 입력 문서와 출력 문서를 명시한다.
4. 의사결정 원칙을 넣는다.
5. 품질 기준과 self-check 를 넣는다.
6. 금지사항을 넣는다.
7. 다음 단계 handoff packet 을 넣는다.
8. review queue 와 changelog 를 갱신한다.

## 자주 발견되는 문제

### 문제 1. description 이 너무 추상적임
- 나쁨: `기능 관련 작업을 수행하는 스킬`
- 좋음: `PRD와 UX를 기반으로 기능 규칙, 상태, API 계약을 정리해 docs/specs 문서를 작성하는 스킬`

### 문제 2. 입력이 없음
- 결과적으로 Skill 이 임의 추정을 하게 된다.
- 해결: 필수 입력과 선택 입력을 분리해서 적는다.

### 문제 3. 출력 파일 경로가 없음
- 결과물이 흩어지거나 누락된다.
- 해결: 갱신 문서와 생성 문서를 경로까지 적는다.

### 문제 4. 역할 경계가 넓음
- 다른 Skill 과 겹치고 책임이 모호해진다.
- 해결: `무엇을 한다`뿐 아니라 `무엇을 하지 않는다`를 넣는다.

### 문제 5. handoff 가 없음
- 다음 Skill 이 맥락을 다시 만들게 된다.
- 해결: handoff packet 을 고정해 둔다.

### 문제 6. 품질 기준이 없음
- 완료 판단이 사람마다 달라진다.
- 해결: self-check 와 완료 기준을 넣는다.

## 개선 후 확인 질문
- 이 Skill 은 어디서 시작하고 어디서 끝나는가
- 입력 없이도 오해 없이 쓸 수 있는가
- 다음 단계가 다시 묻지 않고 이어갈 수 있는가
- 다른 Skill 이 같은 일을 하고 있지 않은가
- 이 Skill 을 처음 읽는 사람도 바로 실행할 수 있는가

## 추천 템플릿
- 역할
- 언제 사용하나
- 입력
- 출력
- 작업 절차
- 판단 원칙
- 품질 기준
- 금지
- 다음 단계 handoff

## 문서화 원칙
- 문장은 짧고 단정적으로 쓴다.
- 파일 경로는 구체적으로 쓴다.
- 가정은 명시하고, 암묵적으로 두지 않는다.
- 예외와 제한을 숨기지 않는다.
