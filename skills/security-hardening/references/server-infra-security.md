# Server and Infra Security Reference

## 목적
- 서버, 컨테이너, 배포 경로, 네트워크 노출을 점검할 때 공통 기준으로 쓴다.

## 1. 네트워크와 노출면
- 공개 포트는 필요한 것만 연다.
- 관리자 포트, DB 포트, SSH, metrics 엔드포인트는 인터넷 직접 노출을 피한다.
- reverse proxy, TLS termination, internal-only 경로를 문서로 설명 가능해야 한다.
- staging 과 production 의 접근 경로를 분리한다.

## 2. 서버 접근 제어
- 운영 접근은 개인 계정 기반으로 남겨야 하고 공유 계정 사용을 줄인다.
- root 직접 로그인보다 제한된 sudo 와 감사 가능한 계정을 우선 검토한다.
- break-glass 접근이 있으면 사용 조건과 로그 남김 규칙이 필요하다.
- SSH 키 회전과 퇴사/권한 변경 시 제거 절차가 있어야 한다.

## 3. 컨테이너와 런타임
- 컨테이너는 불필요한 패키지와 쉘 도구를 줄인다.
- 앱 프로세스는 가능하면 non-root 로 실행한다.
- health check 와 readiness 가 있어야 무한 재시작과 장애 은폐를 줄일 수 있다.
- 디버그 모드, verbose 로그, 개발용 시드 엔드포인트가 배포 환경에 남지 않게 한다.

## 4. 데이터와 백업
- DB 접근 정보는 앱과 운영자에게 최소 범위로 나눈다.
- 백업 파일 저장 위치, 암호화, 보존 기간, 복구 테스트 기록을 확인한다.
- staging 복구 리허설에 production 실데이터를 그대로 쓰는지 점검한다.
- 로그, 파일 스토리지, DB 백업이 서로 다른 보안 경계에 있는지 확인한다.

## 5. 배포와 공급망
- 배포는 재현 가능한 경로를 가져야 한다.
- CI/CD 토큰은 최소 권한과 환경 분리를 따른다.
- dependency update 는 취약점 영향 범위를 적고 적용한다.
- 이미지, lockfile, 배포 스크립트가 drift 없이 맞는지 확인한다.

## 6. 운영 콘솔과 내부 도구
- `/ops`, `/admin`, preflight API 같은 내부 도구는 접근 제한이 있어야 한다.
- 운영 화면은 읽기와 쓰기 권한을 분리한다.
- 위험한 액션은 재확인, 감사 로그, 실행자 식별이 필요하다.
- 내부 도구라도 고객 데이터 bulk export 는 별도 승인 절차가 있어야 한다.

## 7. 장애와 사고 대응
- 비정상 트래픽, 인증 실패 급증, 관리자 접근 시도 증가를 감지할 기준이 있어야 한다.
- 롤백, 시크릿 폐기, 토큰 회수 절차가 runbook 에 있어야 한다.
- 사고 후에는 원인, 노출 범위, 즉시 조치, 재발 방지로 기록을 남긴다.

## 우선순위 힌트
- P0: DB/관리자 포트 공개, shared root access, production secret 노출
- P1: staging-production 경계 붕괴, root 실행, 백업 보호 미흡, internal tool 인증 약함
- P2: 관찰성 부족, 회전 절차 미흡, 취약점 패치 지연
- P3: 운영 문서 정합성 개선
