# Secret and Operations Security Reference

## 목적
- 시크릿 관리, 로그 위생, 배포 게이트, 운영 절차 보안 기준을 정리한다.

## 1. 시크릿 분리
- `staging`과 `production` 시크릿은 절대 공유를 기본값으로 두지 않는다.
- DB, 메일, 스토리지, 관리자 토큰, 서명 키는 환경별로 분리한다.
- `.env.*.example`에는 예시만 두고 실제 값은 넣지 않는다.
- 시크릿 값은 문서보다 secret manager 또는 배포 플랫폼 비밀 저장소를 우선한다.

## 2. 시크릿 수명주기
- 발급 주체, 저장 위치, 사용 서비스, 회전 주기를 추적 가능하게 적는다.
- 유출 의심 시 바로 폐기 가능한 절차가 있어야 한다.
- 장기 토큰보다 짧은 수명과 재발급 가능한 구조를 우선한다.
- 공유 토큰을 유지해야 하면 범위, 만료, 교체 계획을 명시한다.

## 3. 로그와 디버깅 위생
- request/response 로그에 토큰, 쿠키, API key, 개인 연락처, 고객 메모 원문을 남기지 않는다.
- 예외 로그는 스택 트레이스와 민감 데이터 마스킹 규칙을 같이 본다.
- 운영자가 디버그 목적으로 민감 데이터를 직접 복사하지 않도록 runbook 을 둔다.

## 4. 배포 전 게이트
- 환경 변수 검증이 통과하지 않으면 배포하지 않는다.
- 새 공개 링크, 인증 방식, 관리자 기능이 들어가면 보안 검토 메모가 있어야 한다.
- production deploy 전에는 최소 1회 staging 또는 동등 환경에서 smoke 를 거친다.
- 시크릿 변경 배포는 롤백 경로와 회수 계획이 있어야 한다.

## 5. 권한 운영
- 운영자, 개발자, 외주 접근 권한은 역할별로 분리한다.
- 퇴사, 역할 변경, 긴급 차단 시 제거 절차를 문서화한다.
- 사람 기준 접근과 서비스 계정 접근을 분리한다.
- 읽기 전용으로 충분한 역할에 쓰기 권한을 주지 않는다.

## 6. 사고 대응 최소 기준
- 탐지 시점
- 영향 범위
- 폐기한 시크릿/토큰
- 고객 영향 여부
- 재발 방지 액션

## 우선순위 힌트
- P0: 실제 시크릿 커밋, production/staging 시크릿 재사용, 유출 후 미회전
- P1: 로그 마스킹 부족, 관리자 토큰 장기 공유, 회수 절차 부재
- P2: 배포 게이트 문서 부족, 회전 주기 미정
- P3: 예시 문서 정리, 네이밍 개선
