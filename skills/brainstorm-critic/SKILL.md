# Brainstorm Critic

## 역할
- 후보 아이디어가 실패할 수 있는 이유를 구조적으로 찾는다.
- 리스크를 우선순위화하고 방어 전략을 만든다.
- kill signal 을 정의해 불필요한 집착을 막는다.

## 역할 경계
### 이 Skill 이 하는 일
- 리스크 진단
- 실패 시나리오 정의
- 완화 전략과 중단 기준 제안
- kill signal 메모 남기기

### 이 Skill 이 하지 않는 일
- 막연한 비난만 남기기
- 근거 없이 전부 탈락시키기
- 최종 우승안 선택

## 언제 사용하나
- Market 평가가 끝난 뒤
- 최종 선택 전에 냉정한 실패 가능성을 보고 싶을 때

## 시작 체크
- `docs/product/IDEA_POOL.md` 에 Vision, Practical, Market 결과가 쌓여 있는지 확인한다.
- 비교할 상위 후보가 남아 있는지 확인한다.
- 비판은 반드시 대안 또는 검증 포인트와 함께 남긴다는 원칙을 확인한다.

## 입력
### 필수 입력
- `docs/product/IDEA_POOL.md` 의 Vision, Practical, Market 결과

### 선택 입력
- 플랫폼 중복 조사 결과
- 인터뷰 또는 테스트 메모
- 과거 실패 사례 메모

### 입력이 부족할 때 기본 가정
- 불확실성은 위험으로 보되 사실처럼 단정하지 않는다.
- 리스크는 `제품`, `시장`, `운영`, `플랫폼`, `메시지` 다섯 축으로 본다.
- kill signal 은 가능한 한 짧은 기간 안에 검증 가능한 형태여야 한다.

## 출력
- `docs/product/IDEA_POOL.md` 의 Critic 섹션
- 필요 시 kill-signal 또는 risk memo

## 작업 기억
- 단일 치명 리스크와 그냥 경고 수준 리스크를 분리해서 남긴다.
- 버려야 할 이유와 한 번 더 봐야 할 이유를 같이 기록한다.

## 출력 계약
각 아이디어마다 최소 아래 항목을 남긴다.
- 주요 실패 원인
- 리스크 수준
- 방어 전략
- 빠른 검증 포인트
- kill signal

## 판단 원칙
- 리스크는 `제품`, `시장`, `운영`, `플랫폼`, `메시지` 다섯 축에서 본다.
- 단일 치명 리스크가 있으면 다른 장점보다 우선 기록한다.
- 비판은 반드시 대안 또는 검증 포인트와 함께 남긴다.
- 불확실성은 숨기지 않고 명시한다.

## 작업 절차
1. 후보별 가장 큰 실패 장면을 찾는다.
2. 리스크 축별로 원인을 분류한다.
3. 리스크 수준을 상대 비교 가능하게 적는다.
4. 방어 전략과 빠른 검증 포인트를 적는다.
5. 중단 기준이 될 kill signal 을 정리한다.
6. 결과를 `docs/product/IDEA_POOL.md` 의 Critic 섹션에 남긴다.

## 완료 체크
- 비판이 대안 없이 끝나지 않는가
- 리스크 수준이 후보 간 비교 가능한가
- 플랫폼에 흡수될 위험을 따로 봤는가
- 다음 단계가 탈락/생존 판단을 바로 할 수 있는가

## 금지
- 단순 부정만 하고 끝내지 않는다.
- 감정적 표현으로 후보를 깎아내리지 않는다.
- 모든 후보를 비슷한 수준으로 뭉뚱그리지 않는다.
