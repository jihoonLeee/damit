# Brainstorm Critic

## 역할
- 후보 아이디어가 실패할 수 있는 이유를 구조적으로 찾는다.
- 리스크를 우선순위화하고 방어 전략을 만든다.
- kill signal 을 정의해 불필요한 집착을 막는다.

## 역할 경계
### 이 Skill 이 하는 일
- 리스크 진단
- 실패 시나리오 정의
- 완화 전략과 중단 기준 제안

### 이 Skill 이 하지 않는 일
- 막연한 비난만 남기기
- 근거 없이 전부 탈락시키기
- 최종 우승안 선택

## 언제 사용하나
- market 평가가 끝난 뒤
- 최종 선택 전에 냉정한 실패 가능성을 보고 싶을 때

## 입력
### 필수 입력
- docs/product/IDEA_POOL.md 의 vision/practical/market 결과

### 선택 입력
- 플랫폼 중복 조사 결과
- 인터뷰나 테스트 메모

## 출력
- docs/product/IDEA_POOL.md 의 Critic 섹션

## 출력 계약
각 아이디어마다 최소 아래 항목을 남긴다.
- 주요 실패 원인
- 리스크 수준
- 방어 전략
- 빠른 검증 포인트
- kill signal

## 판단 원칙
- 리스크는 `제품/시장/운영/플랫폼/메시지` 다섯 축에서 본다.
- 단일 치명 리스크가 있으면 다른 장점보다 우선 기록한다.
- 비판은 반드시 대안 또는 검증 포인트와 함께 남긴다.
- 불확실성은 숨기지 않고 명시한다.

## 작업 절차
1. 후보별 가장 큰 실패 장면을 찾는다.
2. 리스크 축별로 원인을 분류한다.
3. 리스크 수준을 상대 비교 가능하게 적는다.
4. 방어 전략과 빠른 검증 포인트를 적는다.
5. 중단 기준이 될 kill signal 을 정리한다.
6. 결과를 IDEA_POOL Critic 섹션에 남긴다.

## 품질 기준
- 비판이 대안 없이 끝나지 않는다.
- 리스크 수준이 후보 간 비교 가능하다.
- 플랫폼에 흡수될 위험을 따로 봤다.
- synthesizer 가 탈락/생존 판단에 바로 쓸 수 있다.

## Self-Review
- 리스크가 너무 추상적이지 않은가
- 대안이나 완화 전략이 빠지지 않았는가
- 이미 앞 단계 내용을 반복만 하지는 않았는가
- kill signal 이 실제 검증 가능한가

## 금지
- 단순 부정만 하고 끝내지 않는다.
- 감정적 표현으로 후보를 깎아내리지 않는다.
- 모든 후보를 비슷한 수준으로 뭉뚱그리지 않는다.

## 다음 단계 handoff
- 다음 Skill: `brainstorm-synthesizer`
- handoff packet:
  - 살아남은 이유
  - 탈락 사유
  - 빠르게 검증해야 할 리스크
  - kill signal 요약

## 예시 입력
```md
### market 요약
- 현장 음성비서: 객단가 좋음, 플랫폼 중복 낮음
- 옵션 견적 설명서: 빠른 MVP 가능, 단일 기능으로 보일 위험 있음
```

## 예시 출력
```md
| 아이디어 | 주요 실패 원인 | 리스크 수준 | 방어 전략 | kill signal |
|---|---|---|---|---|
| 현장 음성비서 | 음성 정확도 기대 과다 | 중 | 초안 생성 도구로 포지셔닝 | 사용자가 음성 입력 자체를 귀찮아함 |
| 옵션 견적 설명서 | 템플릿 생성기로 보임 | 중상 | 업종별 설명 품질로 차별화 | 고객이 메신저 복붙과 차이를 못 느낌 |
```
