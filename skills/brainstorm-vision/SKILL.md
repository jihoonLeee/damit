# Brainstorm Vision

## 역할
- 새로운 아이디어를 넓게 발산한다.
- 아직 feasibility, market, risk 기준으로 조기 탈락시키지 않는다.
- 기능보다 문제 장면, 사용자, 기대 가치에 집중한다.

## 역할 경계
### 이 Skill 이 하는 일
- 서로 다른 방향의 후보를 최소 5개 이상 만든다.
- 각 후보가 해결하는 핵심 문제를 선명하게 적는다.
- 왜 이 아이디어가 지금 의미 있는지 가설을 만든다.
- 다음 단계가 검증할 전제를 남긴다.

### 이 Skill 이 하지 않는 일
- 구현 가능성 최종 판단
- 시장성 최종 판단
- 후보 탈락 결정
- 최종 우승안 선정

## 언제 사용하나
- 아이디어가 아직 없거나 너무 좁을 때
- 기존 아이디어를 다시 발산해야 할 때
- PRD 이전 단계에서 선택지를 넓혀야 할 때

## 시작 체크
- 사용자 목표 또는 문제 맥락이 최소 한 문장으로 정리돼 있는지 확인한다.
- 시간, 예산, 운영 제약이 있으면 함께 받는다.
- 지금 단계에서는 탈락보다 발산이 목적이라는 점을 다시 확인한다.

## 입력
### 필수 입력
- 사용자 목표와 배경
- 시간, 예산, 기술, 운영 제약

### 선택 입력
- 타깃 업종 또는 사용자 힌트
- 기존 아이디어 문서: `docs/product/IDEA_POOL.md`
- 이전 선정 결과: `docs/product/IDEA_FINAL.md`

### 입력이 부족할 때 기본 가정
- 사용자는 빠른 검증과 현실적 수익화를 원한다.
- 1인 또는 소규모 팀 기준으로 시작한다.
- 거대 플랫폼과의 정면 충돌은 기본적으로 피하려 한다.

## 출력
- `docs/product/IDEA_POOL.md` 의 Vision 섹션
- 필요 시 `docs/product/IDEA_POOL.md` 안의 발산 기준 메모

## 작업 기억
- 왜 이 문제 장면을 후보로 남겼는지 짧은 근거를 기록한다.
- 바로 탈락시키지 않은 애매한 후보도 다음 단계가 판단할 수 있게 메모를 남긴다.

## 출력 계약
각 아이디어는 최소 아래 항목을 포함한다.
- 아이디어명
- 한 줄 설명
- 핵심 문제
- 핵심 사용자
- 기대 가치
- 왜 지금 의미 있는지
- 다음 단계가 검증할 전제

## 판단 원칙
- 새로운 기능보다 새로운 문제 프레이밍을 우선한다.
- 비슷한 후보라도 타깃이나 가치가 다르면 남긴다.
- 사용자의 수익, 시간, 불안 감소 중 하나와 연결되는 후보를 우선한다.
- 아직 `좋아 보이는가`보다 `문제가 선명한가`를 본다.

## 작업 절차
1. 사용자 목표와 제약을 3~5줄로 요약한다.
2. 후보를 문제 장면별로 최소 5개 이상 만든다.
3. 각 후보를 `문제/사용자/가치` 구조로 정리한다.
4. 중복 후보는 합치되 관점 차이는 보존한다.
5. `brainstorm-practical` 이 검증할 전제를 후보별로 2~3개 남긴다.
6. 결과를 `docs/product/IDEA_POOL.md` 의 Vision 섹션에 기록한다.

## 완료 체크
- 후보가 서로 충분히 다른가
- 기능 목록이 아니라 문제와 사용자가 먼저 적혔는가
- 경쟁이나 난이도로 조기 탈락시키지 않았는가
- 다음 단계 질문거리가 남아 있는가

## 금지
- 구현 난이도만 보고 아이디어를 제거하지 않는다.
- 경쟁 존재만으로 탈락시키지 않는다.
- final 선택처럼 한 후보만 밀어붙이지 않는다.
- 다른 brainstorm skill 의 역할까지 대신하지 않는다.
