# Brainstorm Practical

## 역할
- vision 단계 아이디어를 실제로 시작 가능한 수준으로 현실화한다.
- 자동화가 어려우면 반자동/수동 운영 가능성까지 포함해 본다.
- 제한된 시간과 리소스 안에서 어떤 후보가 살아남는지 판단한다.

## 역할 경계
### 이 Skill 이 하는 일
- MVP 난이도, 개발 기간, 의존성, 초기 운영 방식을 비교한다.
- 실험 가능한 최소 범위를 제안한다.

### 이 Skill 이 하지 않는 일
- 시장성 최종 판단
- 가격 방어 최종 판단
- 최종 우승안 선정

## 언제 사용하나
- vision 단계가 끝난 직후
- 아이디어를 실제 검증 가능한 크기로 줄여야 할 때

## 선행 조건
- IDEA_POOL 에 vision 후보가 정리돼 있어야 한다.
- 사용자의 시간/예산 제약이 있다면 함께 있어야 한다.

## 입력
### 필수 입력
- docs/product/IDEA_POOL.md 의 vision 결과
- 사용자 시간/예산/운영 제약

### 선택 입력
- 기술 스택 선호
- 외부 플랫폼/API 의존 조건

## 출력
- docs/product/IDEA_POOL.md 의 Practical 섹션

## 출력 계약
각 아이디어마다 최소 아래 항목을 남긴다.
- MVP 난이도
- 예상 개발 기간
- 초기 운영 방식
- 필수 의존성
- 막힐 수 있는 지점
- 우회 가능한 수동 운영 방식

## 판단 원칙
- 완벽한 자동화보다 빠른 검증 가능성을 우선한다.
- 4~6주 안에 가치 검증이 가능한지를 핵심 기준으로 본다.
- 기술적 난제보다 운영 복잡도가 더 치명적일 수 있음을 반영한다.
- 너무 큰 플랫폼 연동은 후순위로 미룬다.

## 작업 절차
1. 각 아이디어의 핵심 가치를 정의한다.
2. 그 가치가 나오는 최소 기능 세트를 적는다.
3. 자동화가 막히면 수동/반자동으로 우회 가능한지 본다.
4. 개발 기간과 운영 난이도를 평가한다.
5. 현실적으로 출시 가능한 후보를 표시한다.
6. 결과를 IDEA_POOL Practical 섹션에 정리한다.

## 품질 기준
- 후보별 난이도 판단에 이유가 적혀 있다.
- 수동 운영 가능성까지 검토했다.
- MVP 범위가 실제로 작다.
- market 단계가 바로 이어받을 수 있다.

## Self-Review
- 구현 난이도만 보고 섣불리 불가 판정을 내리지 않았는가
- MVP가 여전히 너무 크지 않은가
- 플랫폼/API 의존성이 과소평가되지 않았는가
- 현실적으로 시작 가능한 후보를 명확히 구분했는가

## 금지
- 시장성 판단을 대신하지 않는다.
- `기술적으로 가능`을 `운영 가능`과 혼동하지 않는다.
- 완성형 제품 기준으로만 평가하지 않는다.

## 예시 입력
```md

### 제약
- 혼자 개발
- 4~6주 MVP
- 외부 API 의존 최소화
```

## 예시 출력
```md
| 아이디어 | MVP 난이도 | 예상 기간 | 초기 운영 방식 | 우회 전략 |
|---|---|---:|---|---|
| 현장 음성비서 | 중 | 4주 | 음성 업로드 + 반자동 구조화 | 초반에는 수동 보정 허용 |
| 시공 리포트 비서 | 중하 | 3주 | 사진 업로드 + 템플릿 리포트 | PDF 생성부터 시작 |
| 옵션 견적 설명서 | 하 | 2주 | 입력폼 + 견적 초안 생성 | 업종 1개만 우선 지원 |
```
