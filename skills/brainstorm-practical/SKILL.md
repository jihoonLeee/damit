# Brainstorm Practical

## 역할
- vision 단계 아이디어를 실제로 시작 가능한 수준으로 현실화한다.
- 자동화가 어려우면 반자동 또는 수동 운영 가능성까지 포함해 본다.
- 제한된 시간과 리소스 안에서 어떤 후보가 살아남는지 판단한다.

## 역할 경계
### 이 Skill 이 하는 일
- MVP 난이도, 개발 기간, 의존성, 초기 운영 방식을 비교한다.
- 실험 가능한 최소 범위를 제안한다.
- 수동 우회 전략을 남긴다.

### 이 Skill 이 하지 않는 일
- 시장성 최종 판단
- 가격 방어 최종 판단
- 최종 우승안 선정

## 언제 사용하나
- vision 단계가 끝난 직후
- 아이디어를 실제 검증 가능한 크기로 줄여야 할 때

## 시작 체크
- `docs/product/IDEA_POOL.md` 에 Vision 후보가 있는지 확인한다.
- 사용자의 시간, 예산, 운영 제약이 정리돼 있는지 확인한다.
- 완전 자동화가 아니라 `검증 가능한 최소 운영`이 목표임을 확인한다.

## 입력
### 필수 입력
- `docs/product/IDEA_POOL.md` 의 Vision 결과
- 사용자 시간, 예산, 운영 제약

### 선택 입력
- 기술 스택 선호
- 외부 플랫폼 또는 API 의존 조건
- 기존 운영 경험 메모

### 입력이 부족할 때 기본 가정
- 혼자 또는 소규모 팀이 4~6주 안에 MVP 를 만들려 한다.
- 초반에는 수동 운영이 일부 허용된다.
- 외부 API 의존은 적을수록 유리하다고 본다.

## 출력
- `docs/product/IDEA_POOL.md` 의 Practical 섹션
- 필요 시 MVP 현실화 메모

## 작업 기억
- 기술적으로 가능하지만 운영이 무거운 이유를 남긴다.
- 탈락보다 `어떻게 더 작게 자를지`를 우선 기록한다.

## 출력 계약
각 아이디어마다 최소 아래 항목을 남긴다.
- MVP 난이도
- 예상 개발 기간
- 초기 운영 방식
- 필수 의존성
- 막힐 수 있는 지점
- 우회 가능한 수동 운영 방식

## 판단 원칙
- 완벽한 자동화보다 빠른 검증 가능성을 우선한다.
- 4~6주 안에 가치 검증이 가능한지를 핵심 기준으로 본다.
- 기술적 난제보다 운영 복잡도가 더 치명적일 수 있음을 반영한다.
- 너무 큰 플랫폼 연동은 후순위로 미룬다.

## 작업 절차
1. 각 아이디어의 핵심 가치를 다시 한 줄로 잡는다.
2. 그 가치가 나오는 최소 기능 세트를 적는다.
3. 자동화가 막히면 수동 또는 반자동으로 우회 가능한지 본다.
4. 개발 기간과 운영 난이도를 평가한다.
5. 현실적으로 출시 가능한 후보를 표시한다.
6. 결과를 `docs/product/IDEA_POOL.md` 의 Practical 섹션에 정리한다.

## 완료 체크
- 후보별 난이도 판단에 이유가 적혀 있는가
- 수동 운영 가능성까지 검토했는가
- MVP 범위가 실제로 작은가
- 다음 단계가 바로 시장성 판단을 할 수 있는가

## 금지
- 시장성 판단을 대신하지 않는다.
- `기술적으로 가능`을 `운영 가능`과 혼동하지 않는다.
- 완성형 제품 기준으로만 평가하지 않는다.
